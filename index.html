What is the MERN stack?
MERN stands for MongoDB, Express, React, Node, after the four key technologies that make up the stack.
  1) MongoDB — document database
  2) Express(.js) — Node.js web framework
  3) React(.js) — a client-side JavaScript framework
  4) Node(.js) — the premier JavaScript web server

API: Application Programming Interface
  API: 
  1) Request 
    1) Protocol: HTTP/HTTPs 
    2) URI
      => URI = Uniform Resource Identifier
      => URI = URL + URN
      => URI = Uniform Resource Locator + Uniform Resouce Name
      Example:
        => https://www.mr-dev.tech/mern
        => URI: https://www.mr-dev.tech/mern
        => URI = URL + URN
          URL = https://www.mr-dev.tech
          URN = mern
        => SERVER IP: 45.87.81.78
        => PING IP: 104.21.6.156 
    3) HTTP Request Method
      => GET: Retrieve Data from data source
      => POST: Save or Process data
      => PUT/PATCH:
        => PUT: Send data for update or create if not existed
        => PATCH: Send data for update and nothing happen if not existed.
      => DELETE: Delete data from datasoure
    4) Request Data:
      => POST,PUT,PATCH
        => Data can be sent to server side using Request BODY
        => Request Body Types:
          => KEY: Value
            => Key always String
            => Value: Can be from any type, converted to string
          => form-data
            => Used to send
              => Files
              => Text
          => x-www-form-url-encoded
            => Used to send
              => Text
    5) Headers
      => Consists of Key & Value
      => Keys & Values are string
      => Examples:
        => Content-type: application/json
        => Accept: application/json
        => Authorization: TOKEN
          => Has an expiration date
          => Unique for each user
          => Can't be duplicated
          => has many types (Bearer)

What is an API?

  1. Modern APIs adhere to standards (typically HTTP and REST), that are developer-friendly, easily accessible and understood broadly
  2. They are treated more like products than code. They are designed for consumption for specific audiences (e.g., mobile developers), they are documented, and they are versioned in a way that users can have certain expectations of its maintenance and lifecycle.
  3. Because they are much more standardized, they have a much stronger discipline for security and governance, as well as monitored and managed for performance and scale
  4. As any other piece of productized software, the modern API has its own software development lifecycle (SDLC) of designing, testing, building, managing, and versioning. Also, modern APIs are well documented for consumption and versioning.
